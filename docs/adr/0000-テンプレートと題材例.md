# ADR-XXXX: <判断の要点を一文で>

## Context

- どんな問題・違和感・制約があったか
- なぜ「設計判断」が必要になったのか
- Android / Compose / ViewModel など、前提条件があれば書く

## Decision

- 今回 **選んだ設計**
- クラス構成、責務の分け方、状態の持たせ方など
- 実装レベルの詳細ではなく「判断そのもの」を書く

## Why not other options

- 検討したが採用しなかった案
- それぞれを **なぜ捨てたか**
- 「将来ダメになる可能性」もここに含めてよい

## Consequences

- この判断によって得られるメリット
- 受け入れるデメリット・制約
- 将来見直すとしたら、どんな条件か

- UI State を sealed class にした理由
- ViewModel に判断を持たせた理由

## 運用方法

- たくさん書いても、 10 ～ 20 個とする。
- 基本的に今のアーキテクチャの説明は「アーキテクチャドキュメント」に書く。
- ADR には、「なぜそのアーキテクチャにしたのか」、「なぜ他のアーキテクチャを捨てたのか」の理由を書く。
- プロジェクト参画時に必読にするのではなく、迷ったときに辞書的に該当する判断がないかを探す。
- 個別具体的な設計判断は「プルリクエストの説明」に書く。
- 未確定の設計判断の議論は「Github Discussion」で行う。
